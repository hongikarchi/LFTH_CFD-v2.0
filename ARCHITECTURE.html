CFD-driven Parametric Sculpture Design Pipeline

작성 2026-05-26 GPU RTX 5070 Ti (16GB) 엔진 DualSPHysics v5.4.3 모델링 Rhino 8 + Grasshopper 최적화 Wallacei X (예정) Repo hongikarchi/LFTH_CFD-v2.0

0. 한눈에 보기

목표: 위 노즐에서 떨어지는 물이 sculpture를 거쳐 바닥 연못에 도달하도록, 밖으로 튀는 물(splash)을 최소화하는 sculpture 디자인을 GA로 자동 최적화.

현재까지 만들어진 것:

  • DualSPHysics CFD 시뮬레이션을 자동 실행하는 파이프라인 (XML 템플릿 → GenCase → GPU 시뮬 → VTK 출력 → splash 분류 → fitness 숫자)
  • Grasshopper에서 호출 가능한 Python 컴포넌트 코드
  • Rhino에서 시뮬 결과를 PointCloud로 시각화하는 IronPython 스크립트
  • matplotlib 기반 애니메이션 GIF 렌더러
  • End-to-end 검증 완료 (cube STL placeholder + 노즐 시나리오, 138초/eval @ 5070 Ti)

남은 것 (사용자 작업): Grasshopper definition 작성 + Wallacei X 연결 + GA 첫 run.

✓ 검증된 사실

  • RTX 5070 Ti CUDA 13.2 + DualSPHysics GPU/CPU 빌드
  • Rhino MCP socket (port 1999) 양방향 통합 — geom 추출, 변형 mesh push, viewport capture
  • Burst velocity profile inlet (velocityfile XML mode 1) + Idp 기반 streamline 추적
  • Per-test JSON → 자동 STL → 시뮬 → Rhino 시각화 → HTML dashboard
  • 6 collider 모듈 인덱싱 (Z 내림차순), per-module rotation/translation/scale transform
  • Fitness: caught/moved/stuck 분리 — inlet 정체 입자 제외하고 실제 catch rate 측정

→ 다음에 할 일

  • 실험 반복: experiments/test_NN.json 작성 → experiment_runner.py 실행
  • Rhino MCP가 listening 중이어야 함 (port 1999)
  • 각 실험: 변형된 collider mesh + 5색 streamlines를 test_NN/* layer로 push
  • experiments.html 자동 갱신 — moved/caught/catch_rate 비교
  • 장기: Wallacei 또는 Python DEAP GA로 parameter space 자동 탐색

iteration 한 번 비용 (Real sculpture 기준)

  • 1회 시뮬: ~40-90초 (timemax + boundary 입자 수에 따라)
  • 전체 cycle (JSON 작성~Rhino push~HTML 갱신): ~1-2분
  • 수동 hypothesis 탐색 100회: ~2-3시간
  • 자동 GA (예: 30 generations × 20 pop = 600 eval): ~10-15시간 overnight

탐색 가능 변수 공간

  • 6 modules × 7 DOF (rx, ry, rz, tx, ty, tz, scale) = 42 design variables
  • + global viscobound, v_inlet → 44 변수
  • 실용 부분공간: translation 3개 + viscobound 1개 + 모듈 scale 6개 = 10 변수 ≈ 적당한 GA 차원

4.6. Rhino MCP 통합

사용자의 실행 중인 Rhino 인스턴스를 직접 조작하는 socket bridge. rhinomcp 플러그인이 Rhino 안에서 listening (TCP port 1999), 외부 Python이 JSON 명령으로 통신.

Protocol

// Request
{"type": "execute_rhinoscript_python_code", "params": {"code": "..."}}

// Response
{"status": "success", "result": {"output": "..."}}
{"status": "error", "message": "..."}

내장 Python (IronPython 2.7 in Rhino 8) 코드를 server-side 실행. RhinoCommon API 전체 접근.

주요 사용 case

스크립트목적
rhino_mcp_probe.pyport + protocol 확인 (초기 검증)
rhino_mcp_doc_inspect.pydoc 단위, layer, 객체 카운트 조회
rhino_mcp_index_modules.pycollider 6 mesh GUID 캐시 (Z 내림차순 정렬)
rhino_mcp_extract_geom.pycollider STL 추출 + nozzle holes 좌표
rhino_mcp_extract_pond_mesh.pyend_point mesh STL + 5 dice-5 nozzle 위치 자동 분산
rhino_cleanup_sim_viz.pysim_*/stream_*/iter_* layer 일괄 제거
rhino_reparent_layers.py특정 prefix layer를 parent 아래로 이동

단위 변환 규칙

  • Rhino doc은 일반적으로 mm 단위
  • DualSPHysics는 SI 단위 (m, s, kg)
  • 추출 시: mm → m (1/1000)
  • Push 시: m → mm (×1000)
  • 변환은 server-side Python에서 수행 (정확도 보존)

제약

  • Rhino 8이 실행 중이어야 함 (포트 1999 listening)
  • doc은 mm 단위 가정 (다른 단위면 변환 인자 수정 필요)
  • Server는 IronPython 2.7 (CPython3 문법 일부 불가)
  • 큰 payload (예: 폴리라인 수백개)는 chunked send 필수 (~70KB per chunk)
  • 응답 출력이 중복되는 경우 있음 (balanced JSON 파싱 필요)

4.7. 실험 Framework (test_NN)

일관성 있는 sculpture 변형 실험을 위한 정형 framework. 각 실험 = 1개 입력 JSON + 1개 결과 JSON + Rhino layer set.

입력 (experiments/test_NN.json)

{
  "test_id": "test_NN",
  "note": "변형 의도 설명 (대시보드에 표시)",
  "global": {
    "v_inlet": 2.0,       // burst velocity (m/s)
    "burst_end": 0.5,     // emission window (s)
    "dp": 0.10,           // particle distance (m)
    "timemax": 10.0,      // sim duration (s)
    "viscobound": 0.2     // boundary friction multiplier
  },
  "modules": [
    {"index": 0, "rotation_deg": [rx, ry, rz],
                  "translation_m": [tx, ty, tz], "scale": s},
    ... // 6 modules total (0..5)
  ]
}

모듈 인덱스 규칙

collider layer의 6 Mesh 객체를 bbox center Z 내림차순으로 0..5 인덱싱. runs/_collider_modules.json에 GUID + 원본 위치 캐시.

  • index 0 = 최상단 shell
  • index 5 = 최하단 shell
  • 이 인덱스는 모든 test_NN 사이에서 일관됨 (caching 덕분)

변형 수학 (server-side)

각 모듈에 대해 affine transform 체인:

1. 원본 mesh 복제 (DuplicateMesh)
2. bbox center c 계산
3. Transform 합성:
   X = Tuser × Tpos(c) × Rz × Ry × Rx × Scale × Tneg(c)

   · Tneg(c): mesh를 origin으로 평행이동 (c 빼기)
   · Scale: uniform scale around origin
   · Rx, Ry, Rz: 축별 회전 (origin 기준)
   · Tpos(c): mesh를 원위치로 되돌림 (c 더하기)
   · Tuser: 사용자 지정 translation (m → mm 1000배)
4. mesh.Transform(X) 적용
5. 6 모듈 모두 변환 후 mesh.Append()로 합본

출력 구조

runs/iter_test_NN/
├── params.json         # 입력 파일 복사본
├── sculpture.stl       # 변형된 collider 합본 (m 단위)
├── velprof.dat         # burst velocity profile (CSV)
├── case_Def.xml        # DualSPHysics 입력
├── Out/
│   ├── data/Part_*.bi4 # native binary
│   └── PartFluid_*.csv # Idp + position per frame
├── trails.json         # Idp → [(x,y,z), ...]
└── result.json         # caught/moved/stuck/catch_rate

Fitness 계산 (3-bucket classifier)

모든 trail (= Idp 1개)에 대해:

z_drop = pts[0].z - pts[-1].z

if z_drop < 1.0:        →  stuck   (inlet 정체, fitness 제외)
elif in_pond_AABB and z < pond_top_z:
                         →  caught  (목표)
else:                    →  splash  (튐)

주요 metric: catch_rate = caught / moved. moved = caught + splash. total = moved + stuck.

Dashboard 자동 갱신

update_experiments_html.py가 runs/iter_test_*/result.json을 모두 스캔하여 experiments.html 재생성. 테스트 추가/수정 시마다 자동 호출 (runner 끝에 hook).

dp0.10 m↓ 정밀 ↑ but 느림 (dp 절반 → 시간 ~8배)
timemax10.0 s자유낙하 + pond 안착 시간. 부족하면 trail이 중간에 끊김
timeout0.10 sframe 저장 간격
viscobound1.0표면 마찰. ↑ 끈적 (수영하듯), ↓ 미끄러움 (튕김 ↑). 0 = 무마찰 (water 슬라이드)

5-5. Fitness 정의 (수정 가능)

현재 분류 (각 trail = Idp 1개):

z_drop = pts[0].z - pts[-1].z

if z_drop < 1.0:                          → stuck    (inlet 정체, 제외)
elif in_pond_AABB and z_final < pond_top_z: → caught  (목표)
else:                                       → splash  (튐)

주요 metric: catch_rate = caught / moved, splash_ratio = splash / total.

다른 metric 가능 (scripts/experiment_runner.py fitness 블록 수정):

  • 물 도달 시간 (얼마나 빨리 pond 채우나)
  • 연못 안 분포 균일도 (XY 표준편차)
  • sculpture 모듈별 contact ratio (각 모듈 touch 여부)
  • 다목적: catch 최대 + 모듈 touch 보장 (Pareto)

8. 용어집

용어뜻 + 우리 프로젝트에서의 역할
SPH (Smoothed Particle Hydrodynamics)mesh 없는 유체 시뮬레이션 방식. 물을 작은 입자 ~12K개로 표현. 자유표면(splash) 정확 — 우리 시나리오에 최적.
DualSPHysicsGPU 가속 SPH 솔버. 5070 Ti CUDA로 4초 시뮬 138초.
dp (initial inter-particle distance)입자 간 거리. 작을수록 정밀, 느림. 우리는 탐색 dp=0.020, 정밀 dp=0.015, refine dp=0.005.
MKDualSPHysics가 boundary/fluid 식별하는 정수 ID. 10=pond, 20=sculpture, 0=floor, 1=fluid (XML에서 정의).
AABB (Axis-Aligned Bounding Box)축 정렬 박스. 우리 pond와 simulation domain을 정의.
GenCaseDualSPHysics 전처리기. XML → particle 배치 + STL 메쉬 import.
PartVTK / PartVTKOut후처리기. DualSPHysics binary 출력 → VTK/CSV. PartVTKOut은 도메인 밖으로 evacuated된 particle만 추출.
mDBC (modified Dynamic Boundary Conditions)boundary 처리 방식. XML Boundary=2. SPH 표준.
VTKVisualization Toolkit format. 3D 데이터 표준. 우리는 legacy binary polydata.
Wallacei XGrasshopper용 NSGA-II 다목적 GA plugin. Galapagos보다 외부 프로세스 호환 좋음.
NSGA-II (Non-dominated Sorting GA II)다목적 최적화 GA. Pareto front 유지. 우리는 단일 objective (splash 최소화)지만 multi 확장 가능.
Pareto front다목적 최적화에서 어느 objective도 더 좋게 못 만드는 design 집합. "최선의 trade-off 선택지".
FitnessGA가 최소/최대화할 숫자. 우리 = splash_ratio (0~1, 작을수록 좋음).
splash_ratio(연못 밖 particle 수) / (전체 fluid particle 수). 시뮬 끝 시점 측정.
IronPython 2.7Rhino 내부 Python 런타임. _-RunPythonScript가 이걸 사용. Python 2 문법만 됨.
CPython 3일반 Python. scripts/ 코드가 사용. Rhino 8의 ScriptEditor도 CPython3 지원.
GHPython 컴포넌트Grasshopper 안에 Python 스크립트 박는 컴포넌트. input/output 핀 자유롭게 설정.
STL3D 삼각 메쉬 파일 포맷. Rhino에서 export, GenCase가 import.
rhinomcpRhino 플러그인 — TCP port 1999에서 JSON 명령 listen. execute_rhinoscript_python_code로 server-side Python 실행.
IdpDualSPHysics가 각 입자에 부여하는 영구 정수 ID. PartVTK CSV의 컬럼. 시간순 trail 재구성에 사용.
Trail / Streamline같은 Idp의 시간순 위치 시퀀스. Polyline으로 시각화. 입자 1개의 궤적.
burst_endvelocity profile의 emission 종료 시간 (초). 0이면 입자 emission 안 됨.
velocityfile / mode=1DualSPHysics inlet XML 옵션 — 외부 CSV에서 시간별 속도 읽음. burst 분사 구현에 사용.
viscobound (ViscoBoundFactor)boundary 입자와 fluid 사이 점성 곱 계수. ↑ 끈적, ↓ 미끄러움. shell surface 마찰 튜닝.
moved / stuck / caughttrail 분류 3-bucket. moved = z drop >1m. caught = pond 안. stuck = inlet 정체 (fitness 제외).
catch_ratecaught / moved — inlet residue 제외한 실제 catch 비율. 주요 fitness metric.
dice-55개 nozzle hole 분산 패턴. 사각형 4 모서리 + 중심. start_point plate 안에서 균등.
test_NN실험 식별자. zero-padded 2자리. experiments/test_NN.json (입력) ↔ runs/iter_test_NN/ (출력) ↔ Rhino test_NN/ (시각화).
module indexcollider mesh 객체의 안정 ID 0..5. Z 내림차순 정렬 (0 = top). _collider_modules.json에 GUID 매핑 저장.
